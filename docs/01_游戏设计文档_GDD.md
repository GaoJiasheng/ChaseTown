# 《追逐 / Chasing》游戏设计文档（GDD）

> 版本 v0.2 · 2026-07-15 · 目标：第一关可玩 Web 3D Demo，质量对齐一线产品原型
> 技术栈：**React + TypeScript + Three.js + WebGL 2** · 平台：**桌面浏览器 + 手机浏览器**

---

## 1. 一句话概念

> 一个被坏人追赶的小孩，在“学校迷宫”里从**前门**一路奔逃到**后门的警察局**，抵达警察处即获胜。

- 类型：俯视视角追逐 / 迷宫逃脱（Top-down Chase / Maze Runner）
- 情绪基调：紧张、心跳、但不血腥（面向大众、可爱写实）
- 单局时长：**60–120 秒**（第一关）
- 首个交付目标：**第一关完整可玩**，三套美术风格可在运行时切换
- 产品形态：打开 URL 即玩，不要求安装客户端或游戏引擎

---

## 2. 平台与技术栈

| 项目 | 决策 | 说明 |
|---|---|---|
| 应用框架 | React 19 + TypeScript + vinext/Vite | 页面、HUD、加载态与游戏容器 |
| 3D 引擎 | **Three.js** | 场景、相机、光照、动画、射线检测与资源加载 |
| 渲染 | **WebGL 2** | 桌面与移动浏览器统一渲染路径，必要时提供降级档 |
| 目标平台 | 当前及前一主版本的 Chrome / Safari / Edge；iOS Safari 与 Android Chrome | 同一 URL、响应式布局 |
| 寻路 | 网格/导航图 + A*，碰撞与可行走数据由关卡 JSON 提供 | 手机点击寻址与 AI 追逐共用 |
| 输入 | Keyboard Events + Pointer Events | 统一键盘、鼠标与触控 |
| 运行时资产 | **glTF 2.0 Binary（GLB）+ PBR 贴图** | 浏览器只加载 `public/models/` |
| 源美术 | Blender/DCC 源文件、交换文件、原始贴图、预览与报告 | 只放 `art-source/`，不进入网页请求 |
| 版本管理 | Git；大体积制作源使用 Git LFS | 运行时文件仍需压缩和控制下载体积 |
| 托管 | `.openai/hosting.json` 对应的 Web 托管 | 构建产物由根项目发布 |
| 分辨率适配 | CSS 安全区 + 动态相机距离/FOV | 刘海屏、横竖屏和不同宽高比 |

### 2.1 纯 Web 边界

- Web 应用入口固定在仓库根目录的 `app/`、`public/`、`package.json` 与 `.openai/`。
- 浏览器只允许请求 `public/models/` 下的 GLB、贴图和关卡数据。
- `art-source/` 是制作源；`.blend`、`.fbx`、高分辨率原图等不得被应用直接加载。
- `tools/art_pipeline/` 负责从制作源生成、压缩、检查运行时 GLB，并输出可复核报告。
- 仓库不再包含或依赖桌面游戏引擎工程；构建、测试与验收都通过 Node.js 和真实浏览器完成。

## 3. 核心玩法循环

```text
出生在前门 → 观察迷宫 → 躲避/甩开坏人 → 抵达后门（警察局）→ 触发警察保护 → 胜利
                     ↑____________ 被坏人抓到 → 失败 → 重开本关 ______________|
```

- **正反馈**：越靠近警察局，音乐越紧张、光线越暖，安全地标越来越清楚。
- **负反馈**：坏人靠近时，屏幕边缘泛红、心跳音效加快、动态相机适度拉远。
- **无惩罚死亡**：被抓后淡出并立即在起点重开，降低 Demo 阶段挫败感。
- **公平可读**：角色、关键道路、追捕距离和终点方向在常见手机屏幕上都必须清楚可见。

## 4. 操作方案（浏览器双端）

### 4.1 桌面浏览器

- **↑ ↓ ← → / WASD**：始终按屏幕上下左右进行 8 方向移动；对角线归一化，速度不叠加。
- 移动为直接控制，角色朝移动方向平滑转身。
- 松开按键后快速减速停下，保留少量惯性缓冲。
- 鼠标滚轮调整视野；`0` 恢复默认缩放倍率。
- `Esc` 暂停；`R` 重开本关。

### 4.2 手机浏览器

- 默认提供拇指可达的屏幕方向控制，按住即持续移动，松开即停止；方向语义与键盘完全一致。
- 点击可行走地面时，可切换为 A* 自动寻路；再次点击立即改变目标点。
- 点击墙内或不可达区域时，就近吸附到最近可行走点，不报错、不锁死。
- 页面不得因移动手势发生意外滚动或缩放；所有按钮触控热区不小于 44 CSS px。

### 4.3 抽象输入层

> 用统一的 `MoveIntent` 把键盘和触控输入先解释为屏幕坐标，再按固定镜头基向量转换到世界坐标，最后交给同一套模拟器。输入设备不直接修改角色位置；镜头方位与转换基向量必须来自同一个常量，保证桌面与手机共享方向、速度、碰撞、转向和动画逻辑。

## 5. 第一关设计：学校迷宫

### 5.1 空间结构

- 视角：**固定世界方位的俯视 3/4 角**；相机俯角约 55–65°，轻微透视，能看清模型体积、表情轮廓与阴影。
- 相机：角色转身与移动都不得改变世界方位，只允许焦点跟随和距离变化；追逐时根据实际 FOV、宽高比、双方屏幕轴分离量和角色包围边距反算最小安全距离，手动缩放不得低于该距离。
- 地图：学校外墙围合，内部由走廊、教室和操场构成迷宫。
- 尺寸参考：约 40×40 至 60×60 米，保证 60–120 秒跑图时长。
- **前门（起点）**位于地图一端，**后门（警察局，终点）**位于对角另一端。
- 迷宫包含 **2–3 条主路径 + 若干死胡同**，让玩家有路线选择与紧张感。

### 5.2 场景内容（真实感道具，非几何块）

- 走廊：储物柜、公告栏、灭火器、地砖、天花灯。
- 教室：课桌椅、黑板、讲台、散落书本。
- 操场：塑胶跑道、篮球架、草坪、长椅、树木。
- 环境叙事：翻倒的椅子、散落的书包，暗示刚发生追逐。
- 警察局门口：警车、旋转红蓝警灯、暖色门厅光。
- 正式运行时必须使用有完整轮廓、材质和细节的 GLB；基础几何只能用于开发调试，不算交付。

### 5.3 引导设计

- 起点使用冷色和阴影，终点警察局使用暖色高光，天然引导玩家“往亮处跑”。
- 地面可放微弱的磨损痕迹或箭头涂鸦，融入场景，不做突兀的 UI 贴片。
- 首次游玩时，坏人延迟 **1.5 秒**启动，给玩家观察和反应时间。

### 5.4 胜负条件

| 结果 | 触发 | 表现 |
|---|---|---|
| **胜利** | 小孩进入警察局门口终点区域 | 警察上前、坏人退场、短暂停格、胜利音乐与通关面板 |
| **失败** | 坏人接触小孩的碰撞范围 | 画面淡出、心跳停顿音、约 0.8 秒后重开 |

## 6. 反派 AI（坏人追逐）

使用有限状态机（FSM），行为可预期但仍有压迫感：

| 状态 | 进入条件 | 行为 | 退出条件 |
|---|---|---|---|
| `Idle/Delay` | 关卡开始 | 站立/张望，延迟 1.5 秒 | 延迟结束 |
| `Suspicious` | 巡逻时首次看见小孩 | 短暂确认、播放警觉反馈，并让固定镜头预先纳入双方 | 证据持续则 Chase；证据丢失则追查最后位置 |
| `Chase` | 视觉确认完成或进入近身感知范围 | 沿导航图追向小孩的当前/预测位置 | 失去视野且超时 |
| `Search` | 丢失小孩 | 跑向最后已知位置，到达后张望约 3 秒 | 再次看见并完成并行确认后 Chase；搜索超时则巡逻 |
| `Patrol` | Search 超时 | 沿预设巡逻点游走 | 看见小孩 |

- **速度平衡**：坏人速度约为小孩的 0.92 倍，但转向角速度更低，玩家可通过拐弯甩开。
- **视野**：前方扇形视锥 + 近身圆形感知；视线用射线或关卡遮挡数据判断。
- **寻路**：玩家点击寻路与 AI 共享同一份可行走网格/导航图，避免两套规则不一致。
- **难度旋钮**：感知半径、视锥角、速度系数、搜索超时、启动延迟集中在 TypeScript/JSON 配置中。
- **公平性**：坏人不知道玩家全局位置，只依据视野、近身感知和最后已知位置行动。
- **表现解耦**：AI / HUD 的“是否观察到追捕者”只影响信息文案，不能删除其 3D 模型；追捕者在逃跑与失败演出中持续渲染，由真实墙体深度、视锥外画幅和柜门遮罩自然决定玩家肉眼是否看见。
- **重获目标**：从断视线、最后位置、搜索或搜柜状态重新发现玩家时必须经过连续 0.2 秒的并行确认；确认期间原状态继续移动和计时，避免短窥视卡住 AI。若确认中断，新目击点覆盖旧搜索证据并转为前往最后位置，不得失忆回巡逻。确认期间若玩家也能看见追捕者，相机提前执行双人安全构图，禁止在 Chase 首帧才突然跳变。
- **搜柜证据优先级**：已经亲眼看见玩家进柜属于强证据；检查途中一帧短窥视不得清除柜体 ID 或取消开柜动作。只有连续确认成功并真正转入 Chase，才允许结束当前搜柜目标。

## 7. 系统架构（Web 模块划分）

```text
app/
├─ game/
│  ├─ core/              游戏流程、状态机、胜负与重开
│  ├─ input/             键盘/触控/点击寻路 → MoveIntent
│  ├─ player/            移动、碰撞、转向与动画状态
│  ├─ ai/                ChaserFSM、感知、巡逻与 A*
│  ├─ camera/            俯视跟随、动态视野、边界钳制
│  ├─ level/             关卡数据、碰撞、起点与终点区域
│  ├─ art/               GLB 加载、角色风格与材质档
│  ├─ ui/                HUD、暂停、胜负与危险反馈
│  ├─ audio/             音乐层、心跳、脚步与警笛
│  └─ config/            难度、风格、资源清单和关卡配置
├─ chasing-game.tsx      游戏容器与生命周期
├─ page.tsx              页面入口
└─ globals.css           响应式布局和安全区

public/models/
├─ characters/           角色运行时 GLB；风格变体由 manifest 映射
├─ environment/          模块与道具运行时 GLB
└─ SharedTextures/       必须外置时使用的运行时共享纹理

art-source/
├─ Concepts/             定稿与返工概念图
├─ Characters/           三角色当前 Blender 母版与必要贴图
├─ Environment/          29 个模块、道具与示例关卡 FBX
├─ _Shared/Animations/   共用骨骼与动画制作源
└─ _Source/              仍需保留的第三方许可证与来源清单

tools/art_pipeline/      导出、优化、压缩、验证和报告脚本
```

**核心解耦原则**

- 玩法逻辑不依赖具体 Mesh、材质或模型文件，只依赖 `CharacterView` 接口。
- 角色逻辑与 Three.js 对象生命周期分离；卸载关卡时正确释放 geometry、material 和 texture。
- 换风格只替换角色视图与渲染配置，移动、碰撞、AI 和胜负逻辑不变。
- 资源 URL 由集中式 manifest 管理，不在游戏循环中散落硬编码路径。

## 8. 三套美术风格切换系统

> 小孩、坏人、警察各提供 **3 套风格**：写实卡通、完全写实、盲盒潮玩；运行时一键切换。

### 8.1 一套骨骼，三套外观

- 三套风格共用同一骨骼层级、骨骼命名和动画 clip 语义。
- 运行时模型均为 GLB，并通过 Three.js `GLTFLoader` 加载。
- 多实例或克隆带骨骼角色时使用安全的 skinned-mesh 克隆方式，禁止直接浅拷贝骨骼引用。
- 角色资源定义示例：

```ts
type ArtStyle = "stylized" | "photoreal" | "blindbox";

type CharacterDefinition = {
  id: "kid" | "villain" | "police";
  variants: Record<ArtStyle, { modelUrl: string; materialProfile: string }>;
};
```

### 8.2 全局风格开关

- `ArtStyleProfile` 使用 TypeScript 对象或 JSON 配置，不依赖编辑器专有资产。
- 风格切换流程：预加载目标 GLB → 保留角色逻辑状态与世界 Transform → 原子替换视图 → 释放旧资源。
- 开发期可使用调试面板切换；玩家通过设置菜单切换。
- 切换失败必须回退到上一套已加载风格，不允许角色消失或游戏中断。

### 8.3 风格与 WebGL 渲染档配对

| 风格 | 角色质感 | 光照与后处理基调 |
|---|---|---|
| 写实卡通 | 真实材质 + 略夸张比例 | 中等对比、暖调、柔和阴影 |
| 完全写实 | 拟真皮肤、布料与金属 | 更强 AO、真实软阴影、轻胶片颗粒 |
| 盲盒潮玩 | 光滑塑质、圆润轮廓 | 高饱和、柔光、干净背景 |

后处理使用 Three.js/WebGL 可实现的效果链，并提供移动端降级；不能为了效果造成明显掉帧或模型不可读。

## 9. 音频设计

- **自适应音乐**：基础层（跑图）+ 紧张层（坏人靠近淡入），按距离混音。
- **脚步声**：小孩轻快、坏人沉重；按地表类型切换素材可后置。
- **心跳**：坏人进入近身感知范围时淡入，越近越快。
- **警笛/门厅**：接近警察局时用警笛和环境声提示“安全在望”。
- **胜利/失败 Stinger**：短促音效收尾。
- 浏览器首次交互前保持静音，获得用户手势后再恢复 `AudioContext`，避免自动播放策略报错。

## 10. UI / UX

- **极简 HUD**：无血条，一击即败；小地图可后置。
- **危险反馈**：坏人靠近时屏幕边缘径向泛红渐强，同时保持中心游戏画面可辨。
- **暂停菜单**：继续、重开、风格切换、音量。
- **胜利面板**：通关动画、用时、“下一关（敬请期待）”。
- **失败流程**：短暂淡出后自动重开，也可手动点击重开。
- UI 使用响应式 CSS，适配 `env(safe-area-inset-*)`；触控热区不小于 44 CSS px。
- Canvas、HUD 和控制区不得互相遮挡；横屏、竖屏与窗口缩放后都能继续操作。

## 11. 性能与加载目标

| 指标 | 目标 |
|---|---|
| 帧率 | 主流桌面稳定 60 FPS；主流手机目标 60 FPS、中低端不低于 30 FPS |
| Draw Call | 第一关常态 < 150；复用材质、InstancedMesh 与合批 |
| 角色面数 | 遵守 02 文档预算；写实角色必须提供 LOD |
| 可见三角面 | 根据设备档动态控制，避免同时展示全部高模 |
| GPU 内存 | 手机浏览器目标 < 512 MB，单关总内存不超过约 800 MB |
| 首次关键下载 | 首屏与首关关键资源压缩后目标 < 60 MB，非关键风格按需加载 |
| Demo 总体积 | 可缓存运行时资源目标 < 300 MB |
| 启动到可玩 | 目标网络和已定义测试设备上 < 5 秒；显示真实加载进度，不假死 |
| 稳定性 | 控制台无未处理异常，模型/贴图/音频请求无 404，重复重开无持续内存增长 |

## 12. 里程碑（第一关）

| 阶段 | 交付 | 依赖 |
|---|---|---|
| M0 Web 工程搭建 | 根目录应用 + Three.js/WebGL + 输入 + 网格寻路跑通白盒追逐 | 无 |
| M1 正式资产接入 | 三套角色和学校环境导出为 GLB，运行时可见，风格切换可用 | 02 文档资产 |
| M2 玩法完善 | AI 状态机、胜负、动态相机、音频与 UI 全部到位 | M0 |
| M3 表现与性能 | 材质、光照、后处理、动画过渡、加载和内存优化 | M1 + M2 |
| M4 浏览器验收 | 桌面 Chrome/Safari + iPhone Safari + Android Chrome 跑通第一关 | M3 |

> **开发顺序**：玩法白盒与美术制作可以并行，但 M1 验收必须使用正式 GLB；几何占位体只能作为开发手段，不能作为“可玩版本”的最终交付。
