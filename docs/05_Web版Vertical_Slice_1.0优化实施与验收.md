# 《追逐 / Chasing》Web 版 Vertical Slice 1.0 优化实施与验收

> 版本：v1.0  
> 日期：2026-07-15  
> 适用范围：仓库根目录的 3D WebGL 版本
> 上游依据：`01_游戏设计文档_GDD.md`、`03_验收Checklist.md`  
> 目标：把当前“可操控的 3D 校园原型”打磨成一局完整、公平、有张力、可在桌面端和手机端验收的追逃游戏。

---

## 1. 产品判断

当前 Web 版已经解决以下基础问题：

- 使用真实 3D 角色与校园模块，不再是二维线条或纯几何白盒。
- 小孩、追捕者、警察均已接入，角色有颜色标记和世界空间标签。
- 起点、出口、计时、胜利、失败、重新开始形成了最小闭环。
- 相机已经支持基础扩大视野、滚轮缩放和追逐时动态构图。
- 现有校园模块、地面材质和主要道具已经进入运行场景。

但当前体验仍更接近“带追兵的 3D 迷宫展示”，还没有达到完整追逃游戏的标准。主要原因不是素材数量不足，而是：

1. 追捕的发现、丢失和搜索规则不可读。
2. 校园道具大多只参与展示，不参与碰撞、遮挡或互动。
3. 角色模型没有使用项目内已有的正式动画。
4. 追逐缺少声音、危险反馈和胜负演出。
5. 手机操作与窄屏构图尚未达到原 GDD 目标。
6. 当前渲染和加载预算超过移动端目标。

因此下一阶段不继续扩关卡或堆模型，优先完成一版高质量 Vertical Slice。

## 3. 本阶段目标与非目标

### 3.1 必须达成

- 玩家能理解“何时被发现、如何甩脱、出口在哪里”。
- 追捕者具备可读、可反制的状态机，不隔墙作弊。
- 现有角色使用真实动画，移动不再像模型滑行。
- 校园区域和道具同时服务于美术、导航和玩法。
- 追逐具有完整的视觉、声音和胜负反馈。
- 手机端具有适合触屏的移动方式和稳定构图。
- 运行态达到明确的帧预算、加载预算和测试门槛。

### 3.2 本阶段不做

- 不制作第二关。
- 不增加第二个追捕者或战斗、武器系统。
- 不做程序化迷宫。
- 不做复杂钥匙、物品栏和大量收集任务。
- 不做多人、账号、云存档和在线排行榜。
- 不继续引入中间版、淘汰版或未验收角色模型。
- 不同时混用三套美术风格；先把一套主风格做到完整自洽。

三套风格切换仍保留为 GDD 主线能力，但排在核心追逃闭环稳定之后。

## 5. P0-A：公平追捕与 AI 状态机

### 5.1 状态设计

Web 版统一为以下状态：

```text
SpawnDelay → Patrol → Suspicious → Chase → Search → Patrol
                         ↑           ↓
                         └── 再次发现 ──┘
```

| 状态 | 行为 | 退出条件 |
|---|---|---|
| `SpawnDelay` | 站立/张望，玩家享有完整保护期 | 1.5 秒结束 |
| `Patrol` | 沿巡逻点移动 | 视锥发现或近身感知 |
| `Suspicious` | 停顿并朝可疑方向观察 | 确认目标或疑虑消失 |
| `Chase` | 追向玩家最近一次可见位置 | 持续丢失视野 |
| `Search` | 到最后已知位置并搜索 | 再次发现或搜索超时 |

### 5.2 感知规则

- 远距离发现必须同时满足：距离、前向视锥、无遮挡。
- 近身圆形感知仍需要基础遮挡检查，避免隔墙贴脸发现。
- 视线使用网格射线或二维线段与阻挡单元相交检查。
- `Chase` 期间只在看见玩家时更新最后已知位置。
- 丢失视野后不再读取玩家实时坐标。
- 镜头和危险反馈读取 AI 状态，而不是只读取双方距离。

### 5.3 寻路与时序

- 游戏逻辑使用固定步长，建议 60Hz。
- AI 感知和寻路以 5–10Hz 更新。
- 仅在目标格改变、状态改变或缓存路径失效时重算 BFS。
- 缓存整数格路径和下一个 waypoint，避免每帧创建 Map、字符串和数组。
- 页面隐藏、窗口失焦或暂停时停止逻辑更新并清理按键状态。

### 5.4 公平性修复

- `SpawnDelay` 期间同时禁用追捕移动和抓捕判定。
- 两条开局主路线均不得在玩家没有决策时间时进入必死区域。
- 玩家速度保持略快于追捕者，但追捕者转向更慢。
- 抓捕使用角色半径/胶囊距离，不使用无体积点坐标。

### 5.5 验收

- [ ] 保护期内任何输入都不会触发失败。
- [ ] 墙后玩家不会被远距离发现。
- [ ] 玩家可以通过连续两个拐角稳定断追。
- [ ] `Search` 只前往最后已知位置，不读取玩家实时位置。
- [ ] 状态切换在 HUD、动画、声音和镜头上保持一致。
- [ ] AI 寻路不再随显示器 60/120Hz 刷新率翻倍执行。

## 6. P0-B：路线决策、碰撞与藏身

### 6.1 路线结构

保留现有 25 × 25 地图，不依靠单纯扩大面积延长时长。将三条主路径定义为：

- **短而危险**：操场/开放走廊，路径短、视线长、缺少藏身点。
- **中等路线**：教室和走廊混合，风险与距离平衡。
- **长而安全**：拐角多、有储物柜，但容易走错和绕远。

死胡同必须至少承担一种作用：藏身、误导、道具叙事或风险回报，不能只是无意义惩罚。

### 6.2 道具玩法化

| 素材 | 玩法职责 |
|---|---|
| 储物柜 | 本阶段唯一主动互动：进入/离开藏身 |
| 课桌椅、讲台 | 阻挡和绕行，形成追逐转弯点 |
| 长椅、树、灌木 | 室外路线遮挡和风险边界 |
| 黑板、公告栏、灭火器 | 区域辨识和方向记忆 |
| 散落书本、书包 | 环境叙事和弱引导 |
| 警车、警局、警灯 | 远近都可识别的终点地标 |

储物柜规则：

- 未被看见进入时，可切断追踪。
- 被追捕者目击进入时，追捕者会到柜前搜索。
- 藏身不是永久安全；停留过久会增加被检查概率或触发提示。
- 只做一个清楚、可靠的藏身机制，不扩展复杂互动系统。

### 6.3 导航信息

- HUD 距离改为最短可行走路径距离，或取消距离、改用方向指示。
- 小屏手机不得隐藏全部出口信息。
- 出口离屏时提供边缘方向标记。
- 起点冷色、出口暖色/警灯、地面磨损和散落物形成无文字引导。

### 6.4 验收

- [ ] 阻挡型道具具有碰撞，玩家和追捕者均不能穿过。
- [ ] 道具不会完全封死合法路线。
- [ ] 三条主路线具有可感知的风险与距离差异。
- [ ] 储物柜可以可靠完成一次断追，并具有被目击后的反制。
- [ ] 出口方向在桌面宽屏、手机竖屏和横屏都始终可读。
- [ ] 首次玩家单局中位数达到 60–120 秒；熟练玩家可达到 30–60 秒。

## 7. P0-C：正式动画、声音与胜负演出

### 7.1 动画接入

使用 `art-source/_Shared/Animations/` 现有资源：

- `Idle`
- `Walk`
- `Run`
- `LookAround`
- `ScaredCaught`
- `Celebrate`
- `PointAlert`
- `TurnLeft`
- `TurnRight`

实施要求：

- 将动画重定向并烘焙到 Web 可直接加载的 GLB，或输出共享动画 GLB。
- 使用 `THREE.AnimationMixer` 管理状态，不再以程序摆臂作为正式方案。
- `Idle/Walk/Run` 之间使用 0.12–0.2 秒 cross-fade。
- 转向角较大时触发转身动作；避免角色瞬间旋转。
- 动画速度与实际移动速度匹配，避免脚滑。
- 警察在远处可使用低频动画更新或冻结姿态。

### 7.2 声音与危险反馈

- 基础音乐和紧张层按 AI 状态与距离混合。
- 追捕者进入 Chase 时播放短促发现音。
- 近距离心跳随危险程度提高音量和节奏。
- 小孩与追捕者使用不同脚步重量。
- 接近出口时逐渐出现警笛/警局环境音。
- 失败和胜利使用独立 Stinger。
- 浏览器第一次点击“开始逃跑”时解锁 WebAudio。
- 提供静音按钮并记住本机设置。

### 7.3 胜负演出

胜利：

1. 小孩进入出口。
2. 警察上前并指示安全区域。
3. 追捕者停止、犹豫并退场。
4. 镜头短暂收紧，播放胜利音乐。
5. 显示本局时间和本地最佳时间。

失败：

1. 触发抓捕动作和危险边缘峰值。
2. 心跳停顿/失败音效。
3. 画面快速淡出。
4. 2 秒内恢复到可操作状态。

### 7.4 验收

- [ ] 三个角色均不再使用滑动式位移观感。
- [ ] 动画状态与 AI/关卡状态同步。
- [ ] 追逐开始和结束仅凭声音与画面即可感知。
- [ ] 胜利包含警察保护和追捕者退场，而不只是弹窗。
- [ ] 失败到再次可操作不超过 2 秒。

## 8. P1-A：墙体、区域化布景与遮挡处理

### 8.1 墙体拓扑

当前实现把每个不可行走格都放置为墙体模块，并把模块强制适配为接近 2 × 1.12 × 2 米体积，容易破坏原模型薄墙比例并形成重复的白色块面。

调整为：

- 墙体沿“可行走格与阻挡格的边界”布置，不填满所有阻挡格。
- 保留源模型长宽厚比例，只按目标宽度和高度等比缩放。
- 优先移除 250 个不邻接走廊的深层墙格渲染。
- Corner、End、Door 按连接拓扑选择，不用方块填洞。
- 重复模块允许实例化，但按材质和阴影规则合批。

### 8.2 区域化布景

- 教室、操场、草地、普通走廊使用明确但统一的材质和色温。
- 道具按功能成组出现，不采用“每种素材只摆一个”的资产清单式陈列。
- 起点保持冷色与阴影；出口区域使用暖光、红蓝警灯和明显轮廓。
- 使用现有散落书本、书包和翻倒家具强化追逐刚发生过的环境叙事。

### 8.3 遮挡与镜头关系

- 相机到玩家之间的墙体使用淡出、切顶或 dither，不继续无限拉远镜头。
- 玩家、追捕者和出口标记不可被近处墙体长期完全遮挡。
- 敌人未被玩家发现且处于墙后时，不通过镜头强行暴露位置。
- 常驻人物标签在新手引导后降低存在感，避免长期呈现调试 UI 观感。

### 8.4 验收

- [ ] 墙体不再呈现大面积厚重白块或屋顶状重复。
- [ ] 墙、门和转角连接无明显拉伸、穿插和悬空。
- [ ] 三个校园区域在 2 秒内可以通过色彩和道具辨认。
- [ ] 玩家处于墙后时仍然可读，且不需要把人物缩得很小。

## 9. P1-B：状态驱动镜头与移动端输入

### 9.1 镜头模式

| 模式 | 构图目标 |
|---|---|
| 探索 | 玩家偏离画面中心，给前进方向更多空间 |
| 怀疑 | 轻微扩大视野，但不暴露墙后的追捕者 |
| 追逐 | 仅在追捕者可见/已发现玩家时尽量纳入双方 |
| 搜索 | 回到以玩家为中心的紧张构图 |
| 胜负 | 使用短时收紧镜头突出角色演出 |

- 保留滚轮缩放和 `0` 重置。
- 增加手机双指缩放或 `+ / -` 镜头按钮。
- 对镜头缩放设置舒适区，避免窄屏自动退到人物过小。
- 对相机焦点和距离分别使用阻尼，避免追逐状态切换时跳变。

### 9.2 手机输入

优先顺序：

1. 点击可行走地面自动寻路，可随时改点。
2. 虚拟摇杆作为连续移动备选。
3. 四方向按钮只保留为无障碍/兼容方案，不作为默认主操作。

适配要求：

- 使用 `env(safe-area-inset-*)` 处理刘海和 Home Indicator。
- 横屏与竖屏分别布局 HUD 和控制区。
- 指针离开、页面失焦、系统手势打断时清空移动状态。
- 控件触摸热区至少 44pt。

### 9.3 验收

- [ ] 390 × 844 竖屏下人物、追捕者状态和出口方向均可辨认。
- [ ] 844 × 390 横屏下 HUD 不覆盖主要路线。
- [ ] 点击寻路可以中途改向，点击墙体会吸附到最近可行走点。
- [ ] 触摸中断后不会持续移动或出现“粘键”。

## 10. P1-C：渲染、加载与工程质量

### 10.1 渲染预算

- 方向光阴影相机只覆盖玩家周围约 20–28 米区域。
- 远处墙体和小型静态道具不投射实时阴影。
- 墙体子网格按共享材质合并，降低每个实例的 primitive 数量。
- 角色使用基于屏幕占比的 LOD0/1/2。
- 警察远离镜头时使用低 LOD 或暂不渲染，接近出口时恢复。
- 减少动态点光数量；不重要的灯改为 emissive 或烘焙效果。
- 提供高/中/低三档：DPR、阴影尺寸、远景细节和点光数量联动。

### 10.2 加载预算

- Kid 和 Villain 继续作为阻塞资源，但使用压缩后的 LOD1 首次进入。
- Police 和出口细节按接近区域或空闲时段加载。
- 环境细节限制为 2–3 个并发请求，不一次性 `Promise.all` 全部加载。
- 角色几何使用 Meshopt 或 Draco。
- 角色和环境纹理使用 KTX2/Basis。
- 共享环境材质与纹理注册表，避免不同 GLB 重复创建相同 GPU Texture。

### 10.3 工程拆分

建议结构：

```text
game/
├─ config/          # 关卡与难度配置
├─ simulation/      # 固定步长、状态、胜负
├─ navigation/      # 网格、BFS、LOS、路径缓存
├─ ai/              # Chaser FSM
├─ renderer/        # 场景、角色、相机、灯光、生命周期
├─ assets/          # 加载、缓存、压缩资源清单
├─ input/           # 键盘、触摸寻路、摇杆
├─ audio/           # 音乐、心跳、脚步、Stinger
└─ qa/              # 指标和测试接口
```

React 组件只负责页面容器、HUD 和菜单，不直接承载全部游戏循环。

### 10.4 测试与恢复

单元测试至少覆盖：

- 最短路径与不可达路径。
- 视锥、墙体遮挡和近身感知。
- 保护期禁止抓捕。
- Chase → Search → Patrol 状态转换。
- 储物柜被看见/未被看见两种结果。
- 胜利、失败和重开状态。

运行态测试至少覆盖：

- 必需 GLB 载入成功，失败时有清楚重试界面。
- Kid/Villain/Police 均存在且位置正确。
- 键盘、点击寻路和摇杆输入。
- 桌面、手机竖屏和手机横屏镜头。
- Draw Calls、三角面和首玩资源预算。
- `webglcontextlost` 后显示恢复提示，不保留黑屏。

### 10.5 验收

- [ ] Draw Calls 稳定 ≤ 150 / 帧。
- [ ] 完整场景渲染三角面稳定 ≤ 100 万 / 帧。
- [ ] 首次可玩阻塞资源 ≤ 12 MiB。
- [ ] 中端手机稳定 ≥ 30 FPS；目标设备尽量稳定 60 FPS。
- [ ] 页面后台停更，回到前台可正常继续。
- [ ] 核心玩法回归由自动化测试覆盖，不只检查 HTML 文案。

## 11. P2：复玩性与风格切换

核心闭环通过验收后再加入：

- 本地最佳时间与评级。
- 2–3 套人工设计的巡逻点/临时封路变体。
- 教学提示的首次显示与后续自动隐藏。
- 写实卡通、完全写实、盲盒三套风格的运行时切换。
- 不同风格对应独立色调、阴影和后处理档位。

若试玩后单局仍明显偏短，只增加一个有叙事理由的轻目标，例如“先触发校园报警器，再抵达警察出口”。不引入多把钥匙和复杂收集链。

## 12. Vertical Slice 1.0 总体验收

### 12.1 新手可理解性

- [ ] 新玩家无需外部说明，10 秒内理解出口方向。
- [ ] 新玩家能理解追捕者何时发现自己。
- [ ] 新玩家至少能描述一种断追方式。

### 12.2 追逃公平性

- [ ] 开局 1.5 秒内绝不会被抓。
- [ ] 敌人不会隔墙远距离发现玩家。
- [ ] 玩家能通过连续拐角或储物柜稳定断追。
- [ ] 两条以上路线同时具有通关可能。

### 12.3 节奏与复玩

- [ ] 5 名首次玩家的单局中位数为 60–120 秒。
- [ ] 熟练玩家可通过路线选择进入 30–60 秒竞速区间。
- [ ] 失败后 2 秒内可重新操作。
- [ ] 胜利显示本局时间和本地最佳时间。

### 12.4 视觉与声音

- [ ] 三个角色使用正式动画，无明显脚滑和瞬时转身。
- [ ] 追逐、搜索、接近出口、胜利和失败均有对应反馈。
- [ ] 墙体不长期遮住玩家，镜头不依赖无限拉远。
- [ ] 校园区域具有清楚的视觉身份，素材不是单件陈列。

### 12.5 双端与性能

- [ ] 桌面键盘、手机点击寻路和摇杆均可完成一局。
- [ ] 竖屏/横屏安全区、控件热区和出口方向均合格。
- [ ] Draw Calls、三角面、首次加载和帧率全部达到 §10.5 预算。
- [ ] 必需资源失败、WebGL 上下文丢失和页面切后台均有可恢复行为。

只有以上五组全部通过，才能把 Web 版标记为“第一关可验收”，而不是“3D 原型可运行”。

5. [ ] 重调三条路线的风险、距离和新手引导。
6. [ ] 接入正式动画与状态混合。
7. [ ] 接入声音、危险泛红和完整胜负演出。

13. [ ] 核心闭环通过后再实现复玩变体与三风格切换。
