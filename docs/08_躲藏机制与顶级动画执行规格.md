# Web 3D《追逐校园逃生》躲藏机制与顶级动画执行规格

> 文档性质：不可静默降低的执行合同
> 适用范围：Web 3D 正式运行版本、所有可供评审的动画原型、桌面端与移动端
> 上位路线图：07《Web 版统一优化方案与路线图》
> 优先级：本文件对“躲藏、追捕 AI、角色动画、道具动画、镜头动画和相关验收”的具体规定高于其他规划文档
> 状态：REVIEW；Web Vertical Slice 的实现、Git、Sites v13、真实浏览器回归与运行截图证据已经齐全，等待用户体验验收

## 0. 这份文件解决什么问题

本文件把“加入躲藏”和“动画必须顶级打磨”从方向性意见变成可直接开发、可逐项验收、可阻止降级的制作合同。

最终要得到的不是一个用方块、圆环、线框和程序摆骨骼演示逻辑的技术样例，而是一段完整可信的 3D 游戏体验：

1. 玩家能读懂危险、跑向真实场景中的藏身点并自然进入。
2. 敌人只有在真正看见或获得合理证据时才知道玩家的去向。
3. 玩家成功断开视线后，敌人会到最后可见点确认、搜索、检查可疑区域，而不是透视追踪。
4. 角色身体、柜门、镜头、声音和玩法状态在同一时间线上同步。
5. 所有对外可见的动画原型均使用正式角色、正式蒙皮、正式材质和正式场景，不以简单几何体替代美术。
6. 每个阶段都有入口条件、交付物、自动检查、人工检查和退出门槛；未通过不得宣称完成。

## 1. 权威层级与防偏差规则

### 1.1 权威层级

发生表述冲突时，按以下顺序执行：

1. 用户最新明确决定。
2. 本文件中带 MUST、禁止、红线、Gate 的规则。
3. 07《统一优化方案与路线图》。
4. 其他旧版分析、原型笔记和实现注释。

### 1.2 唯一状态词

需求和任务只允许使用以下状态：

- PROPOSED：建议，尚未进入承诺范围。
- READY：输入齐备，可以开始制作。
- IN PROGRESS：正在制作，尚不可验收。
- BLOCKED：缺输入或质量不达标，停止向下游传播。
- REVIEW：已有完整证据，等待验收。
- DONE：所有自动与人工门禁均已通过。
- REJECTED：明确不做，必须记录原因。

“基本完成”“临时可用”“先这样”“以后再换”均不是有效交付状态。

### 1.3 变更控制

任何会改变以下内容的决定，必须先改本文档，再改代码或资产：

- 玩家或敌人状态机。
- 敌人能获取的信息和证据。
- 难度参数的含义或默认值。
- 必需动画清单、动画命名或同步事件。
- 模型、材质、灯光、镜头的质量基线。
- 性能预算或验收阈值。
- 阶段交付物与退出条件。

每次变更必须记录：变更原因、影响范围、被替代规则、新验收方式、回滚方法。禁止实现先行、文档长期滞后，也禁止为了赶进度悄悄降低阈值。

### 1.4 阻塞处理

如果正式模型、绑定、动画源文件、贴图或场景道具缺失：

1. 将对应条目标记为 BLOCKED。
2. 列出缺少的资产与可接受输入格式。
3. 可以制作内部技术验证，但必须放在 Debug/实验路径。
4. 不得把几何占位、线框、火柴人、程序摆骨骼包装成“动画原型”或“可评审版本”。
5. 不得让内部占位进入默认构建、用户截图、演示视频或部署版本。

### 1.5 基线冻结与变更等级

A0 通过时建立 Baseline v1.0，并记录：规格版本、Git commit、构建编号、资产 manifest 哈希、Gold Standard 构建、参考视频、视觉基准图、目标设备和性能基线。

之后的变更分为：

| 等级 | 范围 | 审批 |
|---|---|---|
| C0 | 错别字、链接、无语义格式修正 | 文档负责人 |
| C1 | 不改变行为与质量门槛的内部重构 | 技术负责人 + QA |
| C2 | 改变交互、状态机、剪辑、资产或性能 | 产品 + 对应领域 + QA |
| C3 | 降低质量、触碰红线、删除能力或平台 | 用户明确批准 |

变更单必须包含：编号、受影响需求 ID、前后差异、玩法/美术/性能/包体影响、测试变化、风险、回滚方式和批准记录。紧急缺陷只能通过关闭新功能或回滚止损，不能借“紧急”降低美术标准。

### 1.6 开始条件 DoR

任务只有同时满足以下条件才可进入 IN PROGRESS：

- 有唯一需求 ID、明确输出和不做范围。
- 正常、失败、取消和竞态路径已定义。
- 使用的正式模型、骨架、材质、道具和源文件已确定。
- 动画的起止姿势、循环、Root Motion、接触点和事件已定义。
- 桌面与移动端验收场景已经指定。
- 性能影响、依赖、风险和回滚方式已记录。
- “自然”“高级”“有质感”已转换为 Gold Standard 与第 11 节的可测指标。
- 不存在未解决的授权、源文件、骨架兼容或访问问题。

缺一项即为 BLOCKED；不得通过制作可见几何占位来绕过 DoR。

### 1.7 角色与验收权

- 产品/创意负责人：决定体验方向、范围与最终审美接受；由用户行使最终验收权。
- 玩法负责人：保证状态机、数值和公平性。
- 技术负责人：保证架构、资产集成、性能和可回滚性。
- 美术/动画负责人：保证模型、材质、绑定、动作和演出质量。
- QA：依据证据判定 Gate，不得单方面降低门槛。

同一执行者可以兼任多个角色，但必须按不同职责留痕。A2 Gold Standard 和 A6 发布候选必须获得用户实际运行验收；A2 未通过前，不得批量生产依赖该样板的动作。

## 2. 产品体验目标与首个闭环

### 2.1 核心体验句

玩家在校园中被坏人追赶，通过拐角、遮挡和藏身点切断视线；坏人依据最后可见位置和现场证据进行搜索；玩家判断时机离开藏身点并继续逃向警察。

### 2.2 第一阶段只做深一个藏身物

首个正式闭环只承诺“可开门的储物柜/衣柜”一种藏身点。它必须达到完整质量后，才复制到桌下、灌木、厕所隔间等类型。

这样做不是削减体验，而是避免多个半成品机制同时扩散。首个储物柜必须覆盖：

- 靠近与交互提示。
- 玩家进入、关门、隐藏待机、有限窥视、退出。
- 敌人目击进入与未目击进入的两种逻辑。
- 敌人走到柜前、观察、开柜检查、发现或未发现。
- 玩家、敌人、柜门、音效、镜头和碰撞的完整同步。

### 2.3 首个可玩闭环

完整闭环为：

玩家探索 → 敌人发现 → 追逐 → 玩家借遮挡断视线 → 进入储物柜 → 敌人到最后可见点 → 搜索或检查 → 敌人离开 → 玩家退出 → 继续逃跑 → 抵达警察或被抓。

只有这条链路从头到尾可重复完成，才算机制已落地。

## 3. 玩家状态机

### 3.1 状态定义

| 状态 | 含义 | 可接受输入 | 主要输出 |
|---|---|---|---|
| FREE | 正常站立、行走、奔跑 | 移动、转向、交互 | Locomotion 动画与脚步声 |
| ENTER_HIDE | 正在进入柜体 | 仅允许取消窗口内的取消 | 锁定交互目标，播放进入序列 |
| HIDDEN | 完全藏好 | 窥视、退出 | 关闭常规移动，降低玩家可见性 |
| PEEK | 有限窥视 | 左右窥视、松开返回、退出 | 限制镜头与暴露范围 |
| EXIT_HIDE | 正在离开柜体 | 无或仅缓冲移动输入 | 播放退出序列并恢复碰撞 |
| CAUGHT | 被敌人抓住 | 重试 | 抓捕演出和失败结算 |
| ESCAPED | 到达安全点 | 下一局、重试 | 获救演出和成功结算 |

### 3.2 进入藏身点

进入必须同时满足：

- 玩家处于 FREE。
- 藏身点可用，未被占用或锁死。
- 玩家与交互锚点水平距离不大于 0.8 米。
- 玩家朝向与交互正面夹角不大于 70 度；移动端可放宽到 85 度。
- 进入路径无静态障碍，柜门活动范围可用。
- 玩家没有处于 CAUGHT、ESCAPED 或不可中断动作。

按下交互后：

1. 记录 enterStartTime、hideSpotId 和进入时玩家位置。
2. 将玩家移动引导到柜前对齐点，不允许瞬移。
3. 从动画事件 bodyInside 起，玩家身体进入隐藏容积，但仍处于 ENTER_HIDE。
4. 动画事件 doorLatched 表示柜门到达关闭角度，并启用与可见门体一致的遮挡/碰撞。
5. 紧随其后的 hideCommit 才将玩家状态切换为 HIDDEN。
6. 从 ENTER_HIDE 开始到 hideCommit 之前被敌人确认看见，均算目击进入。

### 3.3 取消与输入缓冲

- ENTER_HIDE 的前 0.18 秒允许取消；越过 bodyInside 后不可取消。
- EXIT_HIDE 期间缓存最近一次移动方向，恢复 FREE 后平滑接管。
- 连续点击交互不得重复创建动作、重复占用柜体或跳过动画。
- 所有状态切换由动画事件或明确的安全超时驱动，不得只依靠随意 setTimeout。

### 3.4 隐藏与窥视

- HIDDEN 状态不允许自由移动。
- 默认镜头不显示敌人的精确世界坐标，也不得穿墙显示轮廓。
- 玩家可以在有限角度内窥视；建议水平总范围 50 度、垂直总范围 20 度。
- 窥视会造成可读的小幅暴露，但首版不加入无提示的随机暴露判定。
- 若敌人正面对柜门且距离小于检查距离，退出按钮仍可用，但属于高风险玩家选择。
- HIDDEN 不等于无条件无敌：被目击进入或敌人取得明确证据时，敌人可以检查柜体。

### 3.5 退出藏身点

退出必须满足：

- 玩家处于 HIDDEN 或 PEEK。
- 柜门外的退出占位体没有被静态物体阻塞。
- 当前没有正在执行的柜门检查动作。

退出流程：

1. PEEK 先回正，再进入 EXIT_HIDE。
2. 门打开到 exitClear 事件时恢复玩家动态碰撞。
3. 玩家走出到 locomotionReady 事件后进入 FREE。
4. 如果敌人在退出过程中获得有效视线，立即进入 CHASE，但不截断退出动画造成瞬移。

## 4. 敌人感知与追捕状态机

### 4.0 信息隔离红线

代码结构必须让“不能作弊”成为架构事实，而不是开发者自觉：

- PerceptionSystem 可以读取玩家与场景，但只输出经过视线/声音规则验证的 PerceptionSnapshot。
- ChaserBrain 只能读取 PerceptionSnapshot 和自己的认知黑板。
- SearchPlanner 可以读取藏身点的位置、类型、路径和检查锚点，但不能读取 occupancy、occupiedBy、playerIsHidden 或玩家实时坐标。
- 只有 CHECK_HIDE 动画到达 inspect 事件时，交互系统才允许查询该柜体当时是否有人。

敌人认知黑板只允许保存 lastSeenPosition、lastSeenVelocity、lastSeenTime、lastHeardPosition、soundUncertainty、observedHideEntry、inspectedSpots、searchedNodes、alertLevel 和 searchSeed。出现 playerCurrentPosition、playerCurrentHideSpot、nearestOccupiedHideSpot 等字段即判定红线失败。

### 4.1 状态定义

| 状态 | 含义 | 进入条件 | 退出条件 |
|---|---|---|---|
| SPAWN_DELAY | 开局准备 | 关卡开始 | 倒计时结束 |
| PATROL | 按路线巡逻 | 无有效证据 | 看到玩家或收到证据 |
| SUSPICIOUS | 对弱证据产生怀疑 | 短暂可疑视觉/声音 | 证据增强或衰减 |
| CHASE | 已确认玩家并追赶 | 有确认视觉 | 抓到、逃脱或丢失视线 |
| LOST_SIGHT | 丢失视觉的宽限期 | CHASE 中视觉中断 | 重获视觉并完成并行确认后 CHASE，或宽限结束 |
| GO_LAST_KNOWN | 前往最后可见点 | 宽限结束 | 到达，或重获视觉并完成并行确认后 CHASE |
| SEARCH | 围绕证据点系统搜索 | 到达最后可见点 | 重获视觉并完成并行确认后 CHASE、证据耗尽或超时 |
| CHECK_HIDE | 检查一个有根据的藏身点 | 目击进入或搜索证据足够 | 发现已暴露玩家并完成并行确认后 CHASE，或检查结束 |
| CATCH | 抓捕演出 | 距离与路径条件满足 | 失败结算 |
| RETURN_PATROL | 回到合理巡逻路径 | 搜索结束 | 接上巡逻路线 |

### 4.2 视线判定

敌人确认看见玩家必须同时通过：

1. 距离：玩家在当前难度的视距内。
2. 视锥：玩家关键采样点位于水平与垂直视野角内。
3. 遮挡：敌人眼点到玩家头、胸、骨盆三个采样点至少满足最低可见比例。
4. 持续：视觉置信度达到确认阈值，避免一帧闪现触发满追逐。

遮挡射线只能被登记为 Occluder 的真实墙体、门体、大型道具和地形阻断。透明玻璃、网格栅栏等必须在资产清单中显式标记是否阻挡 AI 视觉。

采样权重固定为头部 0.35、胸部 0.40、骨盆 0.25；感知逻辑频率不得低于 30Hz。ObservedHideEntry 至少需要两个连续有效感知 tick，且入口区域本身可见，避免单帧闪烁误判。

不得以单纯的玩家—敌人距离小于某值代替视线判定。

### 4.3 视觉置信度

建议使用 0 到 1 的 sightConfidence：

- 有效可见时按距离、中心视野权重和可见采样点比例上升。
- 完全遮挡时下降。
- 达到 0.65：确认玩家，进入 CHASE。
- CHASE 中低于 0.20：进入 LOST_SIGHT。
- LOST_SIGHT 宽限期内重回 0.40：恢复 CHASE。

首版标准难度建议：

- 正面视野：110 度。
- 外围弱视野：再增加左右各 20 度，但积累速度减半。
- 确认视距：18 米。
- 丢失视线宽限：0.35 秒。
- 搜索时长：5 秒。

数值最终必须通过关卡实测调整，调整后写回第 10 节参数表，禁止只散落在代码中。

### 4.4 最后可见点

处于 CHASE 时，每 0.10 秒更新一次 lastKnownPosition，但只在视线有效时更新。丢失视线后：

- lastKnownPosition 立即冻结。
- 敌人先前往该点，而不是继续读取玩家实时坐标。
- 玩家隐藏后的坐标不得写入敌人黑板。
- 开发调试可以显示最后可见点，默认构建不可显示。

### 4.5 目击进入规则

在玩家从 ENTER_HIDE 开始到 hideCommit 完成的整个窗口内，只要敌人拥有确认视觉，就记录：

- witnessedHideEntry = true
- witnessedHideSpotId
- witnessedAt

随后敌人允许直接前往并检查该柜体。

若整个进入窗口都没有确认视觉：

- witnessedHideEntry 必须为 false。
- 敌人不得读取 hideSpotId。
- 敌人不得直接导航到玩家所在柜体。
- 敌人只能依据 lastKnownPosition、声音和现场搜索规则选择下一步。

这是公平性的核心红线，必须有自动测试。

### 4.6 搜索策略

SEARCH 不应原地盲转，也不能假装搜索却暗中跟踪玩家。标准流程：

1. 到达 lastKnownPosition 后停顿 0.35 到 0.55 秒，播放辨认环境动作。
2. 根据玩家最后速度方向生成前向候选点。
3. 加入附近的拐角、门口和藏身点作为候选，但不给真实玩家所在点额外权重。
4. 用可解释分数排序：路径可达性、与最后移动方向一致度、距离、近期是否检查过。
5. 选择 2 到 4 个节点依次观察。
6. 只有被目击进入、柜门/声音产生明确证据，或柜体被正常搜索策略选中时，才进入 CHECK_HIDE。
7. 搜索超时且无新证据后进入 RETURN_PATROL。

为避免每次完全一样，可使用关卡种子驱动的有限随机；同一测试种子必须能复现。

标准难度的首版候选评分冻结为：

- 30%：与最后可见点的距离合理性。
- 25%：与最后可见速度方向的一致性。
- 20%：是否位于合理的视线遮挡出口之后。
- 15%：声音或可见门体变化等现有证据。
- 10%：是否尚未搜索。
- 固定种子扰动最多 5%，且不能覆盖明确证据优先级。

评分函数严禁读取玩家位置或柜体占用；在感知日志完全相同的情况下，把玩家暗中放入不同柜体，敌人的搜索状态与选择序列必须相同，直到获得新证据。

### 4.7 声音证据

首个闭环只实现三种清晰声音事件：

- 奔跑脚步。
- 柜门快速开合。
- 玩家碰撞或撞到可发声道具。

声音事件包含 position、radius、intensity、timestamp、sourceType，但不包含玩家实时身份状态。敌人只在听觉半径和遮挡衰减规则内收到事件，并进入 SUSPICIOUS 或更新搜索区域。

声音只能提供事件发生点，不能持续暴露玩家。所有会吸引敌人的声音必须有玩家可听见的反馈。

证据统一分为 SeenPlayer、ObservedHideEntry、HeardSound、ObservedDoorMotion 和 CheckedEmpty。ObservedDoorMotion 只代表门发生变化，不代表里面有人；CheckedEmpty 只证明检查时刻为空，不能反推出玩家当前所在柜体。

标准难度初始声源半径：

| 动作 | 半径 |
|---|---:|
| 步行脚步 | 1.5m |
| 奔跑脚步 | 5.0m |
| 进入柜体的门声 | 3.5m |
| 离开柜体的门声 | 4.0m |
| 撞到可发声道具 | 7.0m |
| 柜内窥视轻响 | 0.8m |
| 柜内静止 | 0m |

墙体、关闭门和楼层按声学材质衰减。听觉事件只能携带带误差的事件位置与不确定半径，不得直接携带玩家对象或 hideSpotId。

### 4.8 抓捕条件

抓捕必须同时满足：

- 敌人处于 CHASE，或 CHECK_HIDE 已确认柜中有人。
- 角色间路径未被墙体阻断。
- 距离小于 0.75 米。
- 条件连续成立至少 0.12 秒，避免高速擦肩误抓。

进入 CATCH 后双方进入配对演出锚点，不允许直接隐藏角色或瞬移到失败界面。

## 5. 藏身点数据合同

每个藏身点必须提供：

- id：关卡内唯一。
- type：首版固定为 locker。
- modelNode：真实可见模型节点。
- enterAnchor：进入对齐点与朝向。
- insideAnchor：身体隐藏位置。
- exitAnchor：退出完成位置。
- cameraAnchor：隐藏镜头基准。
- doorPivot：真实柜门转轴。
- doorClosedAngle / doorOpenAngle。
- occupancy：empty、reserved、occupied、checking。
- visualOcclusion：是否对 AI 视线形成遮挡。
- acousticProfile：开关门响度与隔音系数。
- navCheckAnchor：敌人检查站位。
- animationSet：角色和柜门所需动画映射。

启动关卡时必须验证：

- 锚点齐全且在合理距离内。
- 柜门转轴存在。
- 进入、内部、退出占位体不和静态碰撞相交。
- 玩家与敌人的 animationSet 均能解析。
- 同一个柜体不能被两个实体同时占用。

验证失败时该藏身点禁用并输出可定位错误；不得继续运行后在交互中崩溃。

## 6. 公平性、反露营与难度

### 6.1 公平性原则

- 敌人知道的每条信息都必须来自视觉、声音、路径或明确关卡事件。
- 玩家触发风险时必须得到对应的画面或声音提示。
- 失败后玩家能解释“为什么被发现”，不能归因于随机读心。
- 首版不使用“藏久了自动被发现”的隐形计时器。
- 难度提升优先缩短决策时间、扩大搜索覆盖和提高追逐路线质量，不允许直接给敌人透视。

### 6.2 反露营

首版使用软性反露营：

- 总通关时间影响星级或排名。
- 搜索结束后，敌人返回时可能经过邻近巡逻支路。
- 关卡目标需要玩家继续移动。
- 柜内音效和镜头张力逐渐增强，但不伪造敌人位置。

后续可评估呼吸控制、柜门缝隙暴露等机制，但必须独立提案并给出可学习提示，不能直接塞入首版。

### 6.3 难度参数表

| 参数 | 轻松 | 标准 | 困难 |
|---|---:|---:|---:|
| 敌人正面视野角 | 95° | 110° | 120° |
| 确认视距 | 15m | 18m | 21m |
| 视觉确认阈值时间目标 | 0.55s | 0.38s | 0.25s |
| 丢失视线宽限 | 0.50s | 0.35s | 0.25s |
| 搜索时长 | 4.0s | 5.0s | 6.5s |
| 搜索节点数 | 2 | 3 | 4 |
| 敌人追逐速度 / 玩家跑速 | 0.90 | 0.97 | 1.03 |
| 声音感知半径倍率 | 0.80 | 1.00 | 1.20 |

困难模式的速度优势必须配合拐角、障碍和藏身机会验证，不允许形成直线必败。

## 7. 交互、镜头、界面与声音

### 7.1 交互

- 桌面端交互键默认 E，移动端使用拇指可达的情境按钮。
- 只有最合适的一个藏身点显示交互提示，避免多个提示叠加。
- 提示文案按状态显示“躲进柜子”“离开”“向左窥视”“向右窥视”。
- 图标、文字和按键状态必须在动作开始的同一帧响应。
- 交互按钮禁用时需要有原因，不允许点击无反馈。

### 7.2 镜头

- FREE / CHASE：使用固定世界方位的 3/4 俯视跟随；角色转身不得驱动镜头旋转。镜头只能平滑移动焦点和调整距离，追逐时在同一方位纳入双方安全画幅。
- ENTER_HIDE：使用连续运镜跟随身体进入，不瞬切穿过柜门。
- HIDDEN：镜头位于柜内或设计好的窥视位，FOV 56–60 度，必须尊重墙体遮挡。
- PEEK：只在受控弧线移动，水平建议 ±25 度、垂直建议 ±12 度，不允许自由相机穿模。
- EXIT_HIDE：在 exitClear 后平滑回到跟随相机。
- CATCH：使用短促但清晰的配对镜头，不遮蔽关键身体接触。

常规可操作状态的镜头 bearing 误差必须小于 0.1°，键盘与触控的屏幕方向语义不得随 heading 改变。追逐最低距离必须由 FOV、aspect、双方位置和角色边距反算，390 × 844 下合法最大感知间距仍须双人入画，手动放大不得越过该安全距离。搜索链重获目标时，镜头必须在并行确认窗口预构图，CHASE 首个采样即满足双人安全画幅；确认不能暂停原追查移动，确认中断后必须保留最新目击点并继续前往。镜头切换位置跳变不得大于 0.15 米/帧，特殊剪辑旋转跳变不得大于 5 度/帧；任何会改变操作轴的特殊剪辑必须先冻结玩家移动并经视觉验收。

呼吸镜头位移不得超过 1 厘米、旋转不得超过 0.3 度，并受 Reduced Motion 设置控制。相机必须有真实场景避障，不能穿墙、进入头部或通过柜门提供透视。

追捕者的世界模型与 HUD / AI 的知识状态必须解耦：FREE、CHASE、LOST_SIGHT、GO_LAST_KNOWN、SEARCH 与失败演出均保持正式模型可渲染，不得因导航 LOS 或“视野受限”文案把材质淡到 0；墙体、深度和柜门遮罩仍必须自然遮挡，且禁止穿墙轮廓。

### 7.3 状态反馈

追捕状态通过世界内反馈优先表达：

- PATROL：常态脚步和低强度环境声。
- SUSPICIOUS：敌人停步、抬头或侧耳，短促提示音。
- CHASE：明确的角色动作、音乐层和节奏变化。
- LOST_SIGHT：敌人回望，追逐音乐不立即归零。
- SEARCH：搜索动作、脚步方向和有限的环境音线索。
- RETURN_PATROL：音乐和动作逐步回落。

常驻圆环、头顶文字标签、线框视锥只能用于 Debug 模式，不能成为正式画面的主要表达。

### 7.4 音画同步

- 脚步声绑定 footLContact / footRContact 事件。
- 手碰到柜门绑定 handDoorContact。
- 柜门铰链声绑定 doorMoveStart。
- 关门锁扣声绑定 doorLatched。
- 敌人抓到玩家的冲击声绑定 catchContact。
- 同步误差目标不超过 50 毫秒。

## 8. 美术与动画不可降低的红线

### 8.1 可评审原型的最低外观

所有交给用户查看、录制或部署的动画原型必须同时具备：

- 正式比例的蒙皮角色模型。
- 正确绑定的骨骼、权重和角色朝向。
- 正式或达到正式观感的 PBR 材质与贴图。
- 真实场景模型、真实柜体和真实柜门转轴。
- 可读的主光、补光、阴影和色调。
- 与动作匹配的镜头、地面接触和音效。

内部 Blocking 可以简化节奏，但只能使用真实角色和真实道具的低细节版本；不能用方块人、胶囊人、线条骨架或纯几何体代替角色进行对外评审。

### 8.2 禁止项

默认产品画面中禁止：

- 用 BoxGeometry、SphereGeometry、CapsuleGeometry、CylinderGeometry、PlaneGeometry 组合角色或主要场景道具。
- 用 Line、LineSegments、EdgesGeometry、WireframeGeometry 表示角色、柜体或正式环境。
- 用常驻圆环、箭头、头顶标签替代角色辨识、敌意表达或交互设计。
- 用程序正弦摆动少量骨骼代替正式 AnimationClip。
- T Pose、无动作滑行、瞬移转向、手掌悬空、脚底滑动、身体穿柜、柜门穿身体。
- 用未贴图灰模作为可验收的“顶级动画原型”。
- 默认构建显示碰撞盒、导航网格、视线射线和锚点。

允许的例外仅有：

- 不可见的物理碰撞体。
- 仅在 debug=1 或开发构建显示的调试几何。
- 有明确美术设计、材质、生命周期和性能预算的粒子或抽象特效。

例外不得成为角色、建筑、家具和藏身物的替代品。

### 8.3 当前实现的明确整改项

以下内容只能视为技术债，不能进入本规格的 DONE：

- 用 poseRig 或同类程序摆骨骼承担 Idle、Run、Turn、Search 等主动画。
- decorateActor 或同类函数添加的常驻圆环、头顶标签与辨识用点光。
- 角色运行 GLB 没有正式 AnimationClip。
- 柜门没有真实转轴动画，或玩家与柜门各播各的、没有同步事件。

整改完成后，产品路径必须使用 AnimationMixer/AnimationClip 或等价的正式剪辑系统；程序层只负责状态选择、混合、轻量 LookAt/IK 修正和可控的次级运动。

## 9. 模型、骨骼与导出合同

### 9.1 坐标与尺度

- 单位：1 场景单位 = 1 米。
- 世界上方向：Y Up。
- 角色前方向：统一为 +Z 或由导出规范固定为唯一方向；所有角色一致。
- 角色原点位于双脚中点的地面投影。
- 道具原点位于设计枢轴；柜门原点必须在铰链轴。
- 运行时不得用每角色硬编码旋转和缩放修补错误导出。

### 9.2 骨骼

- 共享角色尽量使用同一套骨架和骨骼命名。
- 必需骨骼至少覆盖 root、hips、spine、chest、neck、head、双侧 clavicle、upperArm、lowerArm、hand、upperLeg、lowerLeg、foot、toe。
- 蒙皮权重归一化；单顶点影响骨骼数符合 Web 运行预算。
- 变换冻结，禁止负缩放和未处理的父级缩放。
- 关键动作需要手指表现时，保留必要手指骨；不以整只僵硬手掌接触柜门。

### 9.3 动画导出

- 格式：glTF 2.0 / GLB。
- 制作采样：30 fps；运行时由引擎插值。
- 常规移动剪辑默认 In Place，位移由控制器负责。
- 配对演出可使用受控 Root Motion，但必须由单一系统拥有位移权，禁止动画和控制器重复位移。
- 循环剪辑首尾速度、姿态与根节点必须连续。
- 每个剪辑必须有稳定、唯一、可机器校验的名称。
- glTF Validator Error 必须为 0。
- 不允许缺失贴图、丢失 Skin、无 Joint、空 Animation 或重复剪辑名。

每个 Clip 在 manifest 中必须声明：clipId、role、skeletonId、duration、sampleFps、loop、rootMode、expectedSpeed、fadeIn、fadeOut、interactionAnchor、events、sourceRevision 和 exportChecksum。生产构建不得依靠模糊字符串搜索猜动画；缺失、重名或骨架不匹配必须构建失败。

Walk 与 Run 必须标记 footLDown、footLUp、footRDown、footRUp，并按步态相位混合，不能只按绝对播放时间淡入淡出。循环剪辑不得包含重复尾帧。

## 11. 动画质量指标

### 11.1 运动质量

- 站立不能完全静止；呼吸、重心和目光变化要克制且可读。
- 起步、停步和转向要体现重量，不能以角色朝向瞬切代替。
- 奔跑步幅与游戏速度匹配；标准速度下脚掌落地滑移目标不超过 2 厘米。
- 接地脚在 planted phase 内不得明显旋转漂移。
- 骨盆高度变化连续，头部不得机械上下打点。
- 肩、肘、腕形成自然动作链，手不以僵硬直臂触碰柜门。
- 进入柜体时身体有避让门框和收肩动作。
- 躲藏待机体现压低呼吸、紧张和空间受限，但不做夸张循环。
- 敌人搜索必须有观察目标和节奏，不是左右匀速摆头。

### 11.2 转场质量

| 转场 | 混合目标 |
|---|---:|
| Idle ↔ Walk | 0.12–0.18s |
| Walk ↔ Run | 0.12–0.20s |
| Locomotion → Turn | 0.08–0.15s |
| Chase → LoseSight | 0.10–0.18s |
| Align → EnterHide | 0.08–0.15s |
| ExitHide → Locomotion | 0.10–0.18s |

- 转场位置爆跳不得超过 2 厘米。
- 转场姿态旋转爆跳不得超过 3 度。
- 输入到可见动作响应目标不超过 100 毫秒。
- One-shot 被中断时必须走明确规则，不允许直接回 T Pose 或绑定姿态。

### 11.3 接触与穿模

- 手掌接触柜门的视觉时刻与 handContact 事件误差不超过 1.5 帧。
- 手掌与把手的空间偏差不得超过 2.5 厘米。
- 柜门打开轨迹与手部运动方向一致。
- 脚底穿地不得超过 1 厘米，且不得连续超过 2 帧。
- 玩家身体与硬质柜门、门框不得出现肉眼可见穿透；任何次要穿插也不得连续超过 2 帧。
- 敌人开柜检查时，手、门把、脚步站位和视线目标必须一致。
- 抓捕演出中两角色接触点不得分离漂移。
- 衣物或头发次级运动不能穿过躯干形成明显视觉错误。

循环首尾的脚/骨盆位置差不超过 1 厘米，关键关节角度差不超过 3 度。默认生产运行中 T Pose、模型闪现、反关节和缩放跳变的允许次数均为 0。

### 11.4 表演与镜头

- 玩家、敌人和警察必须有不同的节奏、重心和轮廓语言。
- 所有动作先保证意图清晰，再增加细节；不以高频抖动冒充细腻。
- 镜头必须展示动作意图和接触点，不得把穿模问题藏在镜头外作为验收方式。
- 同一动作必须在游戏默认镜头、近距离审查镜头和移动端构图下都通过。

Gold Standard 采用七项五分制：身体力学、重量感、动作节奏、轮廓可读性、接触可信度、转场连续性、角色表演。每项不得低于 4 分，核心动作平均不得低于 4.5 分；评分记录与具体修改意见一同进入证据包。

## 12. 动画系统实现合同

### 12.1 分层

运行时动画至少分为：

1. Base Locomotion：Idle、Walk、Run、Turn。
2. Full-body One-shot：进入、退出、检查、抓捕、获救。
3. Additive/IK：LookAt、脚底贴地、手部接触的有限修正。
4. Secondary Motion：头发、衣摆等，按性能档位开关。

IK 只能修正小范围接触误差，不能用来挽救错误的主动作、错误站位或错误柜门枢轴。

### 12.2 单一状态源

- 玩法状态机决定“应该播放什么语义动作”。
- 动画状态机决定剪辑、混合和事件。
- 角色控制器拥有常规移动。
- Root Motion 演出期间由明确的演出控制器接管。
- 柜门控制器只响应同步事件，不自行猜测玩家阶段。
- 音频只响应动画/玩法事件，不轮询骨骼高度猜脚步。

同一个位移、旋转或门角度只能有一个拥有者。

### 12.3 失败保护

- 缺剪辑：开发环境抛出明确错误并阻止该功能进入 READY。
- 动画事件缺失：资产校验失败。
- 动画加载失败：不显示绑定姿态角色冒充成功；进入可诊断错误态。
- 网络或解码慢：先显示有设计的加载状态，不让角色以 T Pose 开始游戏。

## 13. 美术表现基线

### 13.1 角色

- 保留现有三名角色模型作为基础，但必须统一尺度、朝向、材质响应和阴影。
- 角色脸、发型、服装轮廓在游戏镜头距离仍可区分。
- 玩家与敌人的辨识依赖轮廓、配色、动作和灯光，不依赖常驻标签。
- 皮肤、布料、鞋和头发至少有合理粗糙度差异。
- 近景不得出现明显法线破损、权重塌陷或眼睛漂浮。

### 13.2 场景和藏身物

- 优先复用已有 26 个环境模型和贴图。
- 储物柜必须具有真实厚度、门缝、把手、铰链逻辑和内部暗部。
- 场景遮挡要服务追逐路线：拐角可断视线，但不能造成无提示卡死。
- 地面材质和脚底阴影必须稳定，避免角色看似漂浮。

### 13.3 灯光和后期

- 主体轮廓在常态、追逐和柜内三种场景都可读。
- 不靠给每个角色挂常驻 PointLight 解决辨识。
- 色调映射、曝光、阴影质量和环境光必须按桌面/移动两档验证。
- 后期效果不能显著遮蔽交互物、制造重影或降低移动端帧率。

## 14. 性能与仓库体积合同

质量优化不得退化为删除美术；应通过资产组织、压缩、LOD 和按需加载保持观感。

### 14.1 运行预算

| 指标 | 桌面目标 | 移动端目标 |
|---|---:|---:|
| 稳态帧率 | 60 fps，P95 不低于 55 | 30 fps，P95 不低于 28 |
| 可见三角形 | ≤ 1.5M | ≤ 500K |
| Draw Calls | ≤ 250 | ≤ 150 |
| 峰值 GPU 纹理/几何估算 | ≤ 700MB | ≤ 350MB |
| 首次可玩压缩传输 | ≤ 25MB | ≤ 20MB |
| 输入到动作反馈 | ≤ 100ms | ≤ 120ms |

首场景之外的环境和高分辨率贴图按需加载。性能数据必须来自真实运行采样，不能只看源文件大小。

### 14.2 资产策略

- Blender、FBX 等建模与动画母版继续保留。
- 运行时只保留经过优化的 GLB/KTX2 等派生资产。
- 同一模型不得以不同文件名重复复制。
- 共享骨架和可复用动画尽量共用资源。
- 使用合理的 Meshopt/Draco、纹理压缩和 LOD，但压缩后必须重新过视觉 Gate。
- 录屏、审查截图、性能 trace 和临时导出作为 CI/发布证据，不写入 Git 历史。
- 删除真正无引用的重复缓存与构建产物，不删除唯一的美术母版。

## 15. 分阶段制作与退出门槛

任何阶段未通过时，下游可以做独立准备，但不得把上游缺陷固化进正式版本。

### A0：资产与技术基线

输入：

- 现有角色、环境、动画源文件和 Web 运行资产。

交付：

- 资产 manifest。
- 骨架、尺度、朝向、材质、剪辑和引用清单。
- 当前运行性能基线。
- 当前债务清单。

退出门槛：

- 所有保留资产有唯一 ID 和用途。
- 三名角色、柜体和门枢轴能在检查场景正确显示。
- 明确哪些现有 9 个动画源可复用，哪些需要重制。
- 没有无法追溯来源的正式运行资产。

### A1：Gold Standard 角色运动基准

交付：

- Kid 的两套 Idle、Walk、Run Start/Loop/Stop、左右 90 度转向。
- Villain 的 Idle、Patrol Walk、左右转向和 Chase Run。
- 正式模型、材质、灯光、脚步声和默认镜头。

退出门槛：

- 所有第 11 节运动、转场和脚滑指标通过。
- 桌面与移动端均达到性能预算。
- 用户可在一段连续可操作场景中审查，不是单独转台。
- 该阶段成为后续所有动画的质量下限。

### A2：玩家完整躲藏序列

交付：

- 真实柜体、柜门与 Kid 进入、隐藏待机、左右窥视、退出。
- 玩家状态机、碰撞、镜头、交互提示和音效。

退出门槛：

- 角色—柜门同步事件完整。
- 连续进入/退出 30 次无卡死、瞬移、重复占用和穿模。
- 390×844 竖屏、844×390 横屏和桌面默认尺寸均可操作。
- 默认画面没有调试几何或几何占位。

### A3：敌人发现、丢失、搜索、检查与抓捕

交付：

- 完整敌人状态机。
- Alert、LoseSight、Search、CheckLocker、Catch 动画。
- 目击与未目击两条可复现路径。

退出门槛：

- 敌人无任何玩家坐标透视读取。
- 第 18 节所有 AI 场景测试通过。
- 检查柜体、抓捕配对演出没有明显穿模和同步错误。
- 玩家能从反馈判断敌人处于追逐、丢失还是搜索。

### A4：警察与结算演出

交付：

- 警察安全区提示、保护/获救和庆祝。
- 成功与失败结算可重试。

退出门槛：

- 不依赖头顶标签也能看懂安全目标。
- 成功、失败状态不会和追逐状态重叠。
- 完整一局可连续重玩至少 10 次。

### A5：整体验收与美术打磨

交付：

- 完整关卡闭环。
- 最终灯光、材质、音频、镜头、UI 和动画混合。
- Debug 与 Production 构建隔离。

退出门槛：

- 第 17、18、19 节全部通过。
- 无 P0/P1 缺陷。
- 用户以实际运行版本验收，不以编辑器截图替代。

### A6：性能、兼容与发布

交付：

- 桌面和移动性能报告。
- 浏览器兼容结果。
- 生产构建和部署检查。

退出门槛：

- 第 14 节预算达标。
- 无丢模型、丢材质、丢动画、加载后 T Pose。
- 部署 URL 通过 HTTPS、首屏加载、刷新和重开验证。

## 16. 自动化门禁

至少建立以下自动检查；文件名可调整，但职责不可删除：

### 16.1 资产检查

- no-placeholder-art：扫描默认产品视觉路径，拒绝用基础几何作为角色、家具和环境替代；仅允许显式登记的 FX 与 Debug 例外。
- no-procedural-character-pose：拒绝用逐帧正弦摆骨、模型缩放抖动或直接关节随机旋转覆盖正式角色剪辑。
- no-debug-visuals-in-prod：生产构建默认禁止 SkeletonHelper、碰撞框、导航线、视线、圆环、头顶标签和辨识用跟随灯光。
- asset-manifest：所有运行资产有 ID、用途、来源、LOD、贴图和引用者。
- required-animation-clips：解析 GLB，验证第 10 节必需剪辑、名称和非空轨道。
- animation-manifest-consistency：验证时长、循环、骨架、Root Motion、Crossfade、来源版本和 checksum。
- animation-events：验证必需同步事件存在、时间顺序合法且接触阈值达标。
- gltf-validator：Error = 0。
- unused-and-duplicate-assets：报告未引用和内容哈希重复的派生资产。

### 16.2 机制单元测试

- hide-player-fsm：所有合法/非法玩家状态转换。
- hide-occupancy：预留、占用、退出、检查和异常恢复。
- enemy-perception-los：视锥、距离、遮挡和采样点。
- hide-evidence：目击进入时可检查指定柜；未目击时不能读取柜体 ID。
- fair-information-isolation：使用完全相同的感知日志，把玩家分别放入不同柜体；获得新证据前，AI 状态与决策哈希必须一致。
- last-known-position：失去视线后坐标冻结。
- search-seed：固定种子下搜索路径可复现。
- catch-conditions：隔墙、擦肩和瞬时距离不触发误抓。

### 16.3 浏览器运行测试

- 游戏加载后至少存在三名正确蒙皮角色。
- AnimationMixer 或等价正式剪辑系统处于活动状态。
- 默认构建无调试圆环、标签、碰撞盒、视线线段。
- 完整执行一次未目击躲藏和一次目击躲藏。
- 键鼠、触控、竖屏、横屏均完成进入和退出。
- 重开与重复一局后无事件重复绑定、内存持续增长或状态残留。

### 16.4 视觉回归

固定以下审查镜头：

- 玩家跑动侧视脚底。
- 玩家进入柜体的手—门接触。
- 柜内窥视构图。
- 敌人丢失视线和搜索。
- 敌人开柜检查。
- 抓捕配对演出。
- 抵达警察的成功演出。

视觉回归用于阻止意外变化，不能替代人工审美验收。

## 17. 动画人工验收清单

每个动作必须逐项回答“通过/不通过”，并附运行录屏时间码：

- 动作意图在静音情况下是否仍然清楚。

- 柜门是否围绕真实铰链运动。
- 身体是否穿门、门框或地面。
- 循环首尾是否可见跳变。
- 与前后动作混合时是否爆跳。
- 镜头是否展示关键动作而不是遮挡问题。
- 音效是否与接触点同步。

- 桌面、竖屏手机、横屏手机是否都可读。
- 是否使用了本文件禁止的几何、线框或标签替代。

任一关键项不通过，该动作保持 REVIEW 或退回 IN PROGRESS。

## 18. 机制验收场景

### S01 未被目击进入

步骤：让敌人进入 CHASE → 借墙体完全断视线 → 在敌人看不见时进入柜体。

预期：

- 敌人前往最后可见点。
- 敌人不知道玩家所在柜体 ID。
- 敌人按搜索策略检查候选位置。
- 若无证据，不得必然直奔玩家柜体。

### S02 被目击进入

步骤：在敌人保持有效视觉时进入柜体。

预期：

- 记录 witnessedHideEntry 和对应柜体。
- 敌人前往该柜体并播放检查动画。
- 开柜时发现玩家，进入配对抓捕。

### S03 断视线后再次暴露

步骤：进入 LOST_SIGHT 后、搜索结束前走出遮挡。

预期：

- 只有重新通过连续视线确认后恢复 CHASE；确认与原追查状态并行，不暂停移动；确认中断时保留最新目击点并转 GO_LAST_KNOWN。
- 若当前为有 witnessedHideEntry 的 CHECK_HIDE，短暂窥视后重新藏好必须继续原柜检查，不能清除柜体证据或取消柜门演出。
- 390 × 844 下确认窗口提前纳入双方；恢复 CHASE 的首帧玩家与敌人中心均在 0.1–0.9 安全画幅。
- 不发生隔墙转向锁定。

### S04 声音诱导

步骤：敌人看不见玩家时制造脚步或柜门声。

预期：

- 敌人调查声音发生点或区域。
- 声音过期后不再持续提供玩家位置。
- 玩家能听到同一风险事件。

### S05 长时间隐藏

步骤：未被目击、无新增声音，持续停留在柜内。

预期：

- 不触发无提示的随机自动发现。
- 敌人完成合理搜索后返回巡逻。
- 排名时间继续累计。

### S06 柜前退出

步骤：敌人位于柜前但尚未检查时主动退出。

预期：

- 退出动画连续。
- 敌人获得视线后进入追逐。
- 不出现玩家和敌人重叠、瞬移或卡死。

### S07 柜体被占用或阻塞

步骤：重复点击、退出点被阻挡、敌人正检查柜体。

预期：

- 状态机拒绝非法转换并提供反馈。
- 占用不会泄漏，后续仍可正常使用。

### S08 成功与失败闭环

步骤：分别完成逃到警察和被抓。

预期：

- 演出、声音、UI 和结算状态一致。
- 重试后所有 AI、柜体、动画和音频状态重置。

### S09 信息隔离回归

步骤：固定时间步、固定随机种子与完全相同的视觉/声音日志，分别把真实玩家放入 A、B、C 三个柜体。

预期：

- 在敌人取得新视觉、声音或亲自检查之前，三次运行的认知黑板、状态转换、搜索候选和动作序列完全一致。
- 任何一次因真实玩家所在柜体不同而提前分叉，都判定为 AI 作弊和 S1 阻断缺陷。

### 20.1 每个 Gate 的证据包

提交 REVIEW 时必须包含：

- 版本 commit。
- 资产 manifest 变化。
- 对应需求 ID 和测试 ID。
- 自动测试结果。
- 桌面、竖屏、横屏运行录屏或截图。
- 关键动作慢放或逐帧证据。
- 性能采样。
- 已知问题清单。

录屏和 trace 作为 CI/发布附件保存，避免仓库膨胀。

### 20.2 功能 DONE

一个功能只有同时满足以下条件才是 DONE：

- 规则写入本文档。
- 正式资产已接入。
- 默认运行构建可用。
- 自动测试通过。
- 人工视觉和交互清单通过。
- 桌面和移动矩阵通过。
- 性能预算通过。
- 没有以 Debug 或几何占位代替正式效果。

### 20.3 项目阶段 DONE

本轮“躲藏与顶级动画”只有在以下全部成立时完成：

1. A0 到 A6 全部通过。
2. S01 到 S09 全部通过。
3. 第 10 节必需动画全部存在并达到第 11、17 节要求。
4. 产品路径不再使用简单摆骨骼承担主动画。
5. 默认画面不再依赖圆环、标签、线条和几何占位表达角色或机制。
6. 玩家可以稳定完成至少一局并愿意立即再玩一局。
7. 用户以最终部署运行态完成审美与体验验收。

### 20.4 缺陷等级

| 等级 | 定义 | 示例 | 规则 |
|---|---|---|---|
| S0 | 无法启动或核心流程中断 | 黑屏、崩溃、无法控制 | 阻断 |
| S1 | 核心体验错误或红线违规 | 人物不可见、几何占位、T Pose、状态锁死、AI 透视 | 阻断 |
| S2 | 明显局部品质问题 | 脚滑、穿插、接触错位、镜头遮挡、声音错拍 | 核心动作必须清零 |
| S3 | 不影响核心体验的轻微瑕疵 | 极端角度的小幅材质或构图问题 | 登记后评估 |

缺陷降级、豁免或改写为“已知问题”都属于变更，不能由实现者自行决定。

### 20.5 回滚条件

出现任一 S0、人物不可见、必需动画缺失、几何占位进入生产、AI 稳定穿墙锁定，或性能相对上一接受版本下降超过 15%，立即：

1. 停止发布并锁定故障构建和证据。
2. 回滚到最后一个通过对应 Gate 的代码、资产、manifest 和缓存版本。
3. 重跑启动、角色显示、躲藏、AI、胜负和性能烟雾测试。
4. 记录根因和重新上线条件。

若上一版本没有新功能，应关闭该功能并保留上一版正式美术体验；禁止回滚到几何占位版本。

## 21. 执行顺序与不做事项

### 21.1 固定执行顺序

1. A0 资产审计与尺度/骨架合同。
2. A1 先做一个达到最终标准的运动基准。
3. A2 做透一个柜体的玩家完整序列。
4. A3 接入公平感知、搜索和敌人完整序列。
5. A4 完成警察与结算。
6. A5 统一灯光、镜头、音效、UI 和动画混合。
7. A6 性能与发布验收。

任何人不得为了展示功能数量而跳过 A1/A2 的质量基准，批量制造低质量动作。

### 21.2 首轮明确不做

- 多种藏身物同时开发。
- 无提示的呼吸 QTE 或随机暴露。
- 敌人开门、翻桌等大规模环境破坏。
- 多敌人协同搜捕。
- 在线多人。
- 以新增机制掩盖基础移动和动画质量问题。

这些可以进入后续路线图，但不能分散本轮对一个完整高质量闭环的投入。

## 23. 最终红线复述

以下三条没有例外：

1. 对外可见的角色、环境和藏身物不能由简单几何线条堆砌。
2. 主动画不能由少量骨骼程序摆动冒充正式 AnimationClip。
3. 敌人不能读取未通过视觉或证据获得的玩家位置。

缺少正式输入时应诚实标记 BLOCKED；质量未达标时应继续打磨；只有证据完整且实际运行通过时才可标记 DONE。
