# 《追逐 / Chasing》Web 版统一优化方案与路线图

> 版本：v3.0
> 日期：2026-07-23
> 范围：纯 Web 3D 十关主题战役（校园、医院、消防站、工厂）
> 目标：把十关追逃战役收敛为公平、紧张、可反制、可复玩且可在桌面与移动端稳定运行的完整体验，并以正式角色、美术、动画和音乐呈现。

## 1. 统一产品目标

核心体验不是单纯比速度，而是：玩家观察环境和追捕者，利用拐角切断视线，判断是否安全进入藏身处，在敌人的末位搜索中承受压力，再选择离开时机并逃向警察。

当前版本先把一类互动做深：可开门的英雄储物柜。它必须同时连接玩家状态、敌人证据、角色动作、柜门动作、镜头、UI、声音和失败判定；不以多个半成品互动稀释质量。十关可以改变主题、迷宫、追捕压力和搜柜强度，但不能改变这套公平性合同。

发布版本必须满足三条底线：

1. 默认运行画面不出现几何角色、线框角色、程序摆骨骼、调试圆环或文字名牌。
2. 敌人不能隔墙看见玩家，也不能在失去视线后读取玩家实时位置。
3. 任何资产缺失、动作缺失或授权不明都必须显式失败，禁止静默换成占位素材。

## 2. 七个维度的优化策略

### 2.1 可玩性

- 完成“探索 → 被发现 → 追逐 → 断视线 → 躲藏 → 搜索/检查 → 离开 → 逃生/被抓”的一局闭环。
- 玩家略快于追捕者，优势来自直线速度；追捕者优势来自路线压迫，不能靠作弊定位。
- 起点有保护期；保护期同时关闭敌人移动与抓捕。
- 出口、储物柜和危险状态始终可读，但 HUD 不展示墙后的敌人精确位置。
- 桌面与触控使用同一移动和交互意图，不产生两套玩法规则。

### 2.2 耐玩性

- 同一地图提供短险、均衡、长稳三类路线，而不是唯一最优解；搜索节点按感知证据生成可复现种子并重排，避免每局固定背板。
- 藏身决策有信息成本：隐藏安全但失去环境信息，窥视可获得信息但重新暴露。
- 被目击入柜和未被目击入柜产生不同后果，形成可学习而非随机的复玩差异。
- 失败面板不泄露敌人全知信息，只提示可执行的改进方向。
- 每关记录精确到 0.01 秒的本地最佳时间，并提供“藏身脱逃、一次目击、极速逃生”三项精通目标与铜/银/金评价；只奖励技巧，不引入数值养成。

### 2.3 难度设计

- 初始基线：每格 2 米；玩家 2.2 格/秒（4.4 m/s），追捕者追逐 1.85 格/秒（3.7 m/s）；视野 7 格、110°；搜索 4.5 秒。巡逻、搜索和搜柜按正式 Walk 步幅降速。
- 新手容错来自保护期、较慢追捕者和明确状态反馈，不来自削弱碰撞或关闭失败。
- 难度只能调整感知距离、确认时间、搜索时长、转向和速度比例，不能赋予敌人透视。
- 藏身并非无条件免死：目击入柜后敌人必须走到对应柜体并完成检查，才可抓到玩家。
- 每次参数变更必须通过固定步长 30/60/120Hz 一致性、断视线、目击/未目击入柜和可达性回归。

### 2.4 机制设计

- 玩家状态：FREE、ALIGNING_HIDE、ENTER_HIDE、HIDDEN、ENTER_PEEK、PEEK、EXIT_PEEK、EXIT_HIDE、CAUGHT、ESCAPED。
- 追捕者状态：SPAWN_DELAY、PATROL、SUSPICIOUS、CHASE、LOST_SIGHT、GO_TO_LAST_KNOWN、SEARCH、CHECK_HIDE。
- 感知层只输出证据；决策层输入中禁止出现完整 PlayerState、隐藏坐标或柜体占用表。
- 视线由距离、视锥和 supercover 网格遮挡共同决定，防止斜角漏视。
- 躲藏交互由柜前 approach、柜内 concealed、朝向和占用状态共同定义。
- 动画与逻辑共享时长合同；进入、关门、隐藏提交、开门、离开不能各自漂移。
- 进柜在关门安全 marker 后结束视觉暴露；出柜在开门 marker 后才开始暴露；探头必须经过开缝/关缝过渡，不能让规则先于画面切换。

### 2.5 交互体验

- 靠近柜体才显示 E 提示；提示同时说明“躲藏/离开”和“按住观察”。
- 触控按钮热区不小于 44 CSS px，PointerCancel、PointerLeave、失焦和页面隐藏均清除按键。
- 相机使用固定世界方位；角色上下左右转向不得改变镜头 bearing 或按键语义，只允许带世界空间死区的焦点跟随、追逐双人构图和距离缩放。
- 相机默认扩大到能同时判断玩家前路和近距离危险；进入追逐时在固定方位下纳入双方构图，断视线后恢复玩家中心，不追踪墙后的实时坐标。搜索链重新发现玩家时开启 0.2 秒并行确认窗口，原追查仍移动和计时；确认若中断则保留最新目击点并继续前往，不清空证据。双方真实可互见时镜头提前构图，保证恢复追逐的首帧双人已经入画。
- 滚轮、+/− 与 0/重置提供人工视野控制；追逐时按 FOV、宽高比、双方分离量和角色边距反算最低安全距离，手动倍率不得把任一角色挤出安全画幅。
- 出口默认只显示路线距离；只有玩家持续无进展或反向移动时，才显示合法最短路的下一转向提示，并以恢复迟滞避免路口闪烁。
- 藏身 HUD 按玩家已知证据计算生还性：优先考虑断视线时间、截获裕量和可搜索半径；所有柜均为高风险时，只引导到遮挡点，不得回退到最近柜。
- AI / HUD 的情报可见性与 3D 世界渲染分离：追捕者模型不得因距离或导航 LOS 直接淡出，只允许真实深度、墙体、画幅和柜门视野自然遮挡。
- 结果面板延后出现，让被抓或获救动作先完整演出；重开必须瞬时复位角色、相机、柜门和 AI。

### 2.6 美术效果

- 使用项目现有正式角色高模、校园模块和道具；不以新画的几何体替代已有资产。
- 角色保持统一尺度、脚底原点、骨骼和动作命名；主体材质使用真实 BaseColor、Normal、AO、Metallic/Roughness 输入。
- 校园使用冷月光、暖室内灯和绿色出口光建立层级；ACES tone mapping 与环境反射统一材质观感。
- 墙体沿可行走区边界实例化，不在全部阻挡格堆实体方块；地面必须使用项目地砖、草地、教室与操场模块。
- 储物柜作为英雄道具制作真实空腔、门轴、百叶、层板、挂钩、铰链、把手、编号和交互锚点。
- 运行资产和可编辑母版分离；历史候选、备份与评审渲染不进入默认构建。

### 2.7 沉浸体验

- 探索与威胁两条同拍音乐 stem 连续混音，危险上升快、下降慢，避免状态切换硬切。
- 浏览器首次有效手势才解锁音频；双轨保持采样长度与播放位置同步，后台恢复后自动校正。
- 追捕状态通过灯光、边缘暗角、UI 颜色和音乐共同表达，但中心操作区始终清晰。
- 隐藏时收窄视野，表现柜门缝和受限观察；窥视恢复少量环境信息并承担暴露风险。
- 胜利由玩家庆祝和警察保护动作完成，失败由玩家受惊/被制伏与敌人抓捕动作完成，不只弹文字框。

## 3. 动画、美术与音乐质量合同

### 3.1 动画

- 所有角色必须使用真实蒙皮和 `THREE.AnimationMixer`。
- 小孩必需：Idle、Walk、Run、TurnLeft、TurnRight、HideEnter、HideIdle、HidePeek、HideExit、Caught、EscapeCelebrate、Interact。
- 追捕者必需：Idle、PatrolWalk、Run、Alert、LostSight、Search、CheckHide、Catch。
- 警察必需：Idle、Run、Alert、Interact、Resolve。
- 每条动作必须有全身骨骼通道、非静态运动、合理时长和固定机位前后视角接触表。
- Idle/Walk/Run 混合 0.12–0.2 秒；移动速度驱动动作播放率；Root Motion 在导出时移除。
- 柜前 90° 原地转身固定为 0.6 秒 smootherstep；180° 必须显式重启两个同向周期，角色 root yaw 与动作标准化时间共用同一曲线。
- 躲藏进入末段必须与 HideIdle 首帧连续；窥视保持低重心；Caught 必须具备清晰的预备、受击/后坐和保持段。

具体帧、事件、门动画和验收规则见 `08_躲藏机制与顶级动画执行规格.md`。

### 3.2 美术

- 三角色运行 GLB 单体小于 12 MiB，具备完整 21 骨 Web 骨架。
- 材质按物理属性区分皮肤、布料、塑料、金属、鞋和柜体烤漆；不得只靠统一纯色和灯光假装细节。
- 英雄储物柜必须保留 DoorPivot、HideAnchor、HandIK、PeekAnchor、CameraAnchor、SearchAnchor。
- 视觉检查必须至少包括：角色正面/背面动作接触表、柜门关闭/开启/内部三视图、最终运行截图。

### 3.3 音乐

- 探索与威胁 stem 共享 101 BPM、4/4、Bb minor、64 beats、48 kHz 双声道时间线。
- 两条 stem 容器时长差不超过 1 ms，解码样本数一致；运行时每 250 ms 检查漂移。
- 成品经 EBU R128 两遍母带；满威胁混音仍保留峰值余量。
- 禁止 oscillator、蜂鸣或临时合成音作为“音乐原型”兜底。
- 构建输入、哈希、响度、峰值、接缝与授权必须进入 manifest 和许可记录。

## 5. 回归与返工闭环

每一轮都按以下顺序执行，发现问题回到对应生产步骤，不降低门槛：

1. 重建角色、柜体和音乐产物，核对输入哈希。
2. 运行 TypeScript/ESLint、生产构建和全部 Node 测试。
3. 验证每个 GLB 可解析、动画齐全、动作确实运动、授权记录存在、无备份文件。
4. 以固定机位渲染动作与英雄柜接触表，人工检查穿地、滑步、跳姿、穿模和门体时间线。
5. 启动生产服务，逐个请求页面、26 个 GLB、两条音频和 manifest，确认状态码、MIME 与字节数。
6. 在实际运行版本中完成开始、移动、断视线、未目击躲藏、目击检查、窥视、离开、失败、胜利、重开和缩放冒烟。
7. 记录截图、运行统计、异常和返工；任何一项失败都回到第 1 步。
8. 只有同一候选版本全部通过才允许提交、推送和部署。

## 6. 当前发布门禁

以下清单是每个候选版本都必须重新执行的发布模板；具体通过数量、Commit 和 Sites 版本记录在当次发布结果中，避免路线图再次固化成过期状态。

- [ ] 生产构建、TypeScript、ESLint 和全部自动化测试同一轮通过。
- [ ] 三角色动作集名称、时长、全身运动、Kid 转身足锁和单体体积通过门禁。
- [ ] HidePeek 低重心、Caught 有清晰冲击、躲藏衔接无跳姿。
- [ ] 英雄柜结构、锚点、六条门动画和美术 / 运行审查通过。
- [ ] 十关 300 场景藏身矩阵重复运行一致，十关引导闭环全部完成。
- [ ] 公平性 FAIR-01：未目击柜体选择不改变敌人决策；所有高风险柜不产生误导箭头。
- [ ] 保护期、墙体遮挡、末位冻结、左右巡视、目击检查、隐藏免近身抓捕均通过。
- [ ] 精确成绩、个人最佳、三项精通目标和旧进度数据迁移通过。
- [ ] 首局必要资产不等待装饰和警察；装饰渐入、警察胜利 fallback、移动端画质预算通过。
- [ ] 音乐源哈希、双轨样本数、响度、峰值、循环接缝和运行时同步通过。
- [ ] 页面、全部运行资产与音频在生产服务中无 404。
- [ ] 运行画面中人物、储物柜、主题材质和 HUD 清楚可见，无几何占位。
- [ ] 重开、页面隐藏恢复、触控取消、WebGL 恢复和镜头复位无状态残留。
- [ ] 发布版本、远端 Git commit、Sites 源码 commit 与部署产物完全一致。

## 7. 后续路线（不阻塞本轮）

本轮闭环稳定后再按价值排序扩展：

1. 经正式采样、编辑和母带处理的主题脚步、织物、柜门与环境 Foley，替换剩余程序化声音原型。
2. 可听见但不精确定位的主动噪声诱导道具。
3. 教室桌下、厕所隔间、灌木等新藏身物，但复用同一证据合同。
4. 主题专属系统事件与难度预设，避免只靠参数和换皮形成差异。
5. 角色 LOD、KTX2 纹理与跨关卡解码缓存；必须先完成 Safari / iOS 兼容回归。

这些项目不得反向破坏公平感知、正式资产和运行质量底线。
